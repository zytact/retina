Phase 19 — Shipping

What Do We Actually Produce?

A research project is not just ideas and experiments -- it is a set of concrete deliverables: code, trained models, result tables, visualizations, and a manuscript. Phase 19 defines exactly what outputs the system produces, organized into code deliverables, run artifacts, and paper components.

The Goal

A researcher reading our paper should be able to reproduce every result, inspect every artifact, and understand every design decision -- all from what we deliver.

What Phase 19 Is and Is Not

Phase 19 is not a new implementation step. It is an inventory of everything the project produces, organized so you can check completeness before submission. Think of it as a packing list for a journal submission: you do not want to discover at review time that you forgot to save the ablation results or did not log a key hyperparameter.

For beginners: If you have ever turned in a school project only to realize you forgot to include a required file, you understand the purpose of this phase. It is the "did we pack everything?" check before the paper is submitted.

Code Deliverables

1. Dataset Loader

A complete PyTorch Dataset class that handles:

  • Loading OCTA images from disk with layer-specific preprocessing
  • Clinical feature loading with learned mask-token imputation
  • Phase 8 generated mask integration (vessel, FAZ, flow-deficit masks)
  • Patient-level stratified cross-validation fold assignment
  • Bilateral eye handling (OD/OS as independent samples, same patient same fold)
  • Data integrity checks (missing files, mismatched dimensions, corrupt images)

2. Training Pipeline

The full staged training schedule:

  • Phase 3: OCTA-SimCLR pretraining with bilateral positives, per-layer encoders
  • Phase 8C/D: U-Net++ segmentation student training on classical pseudo-labels
  • Phase 12: three-stage disease classifier training (frozen backbones -> fine-tune)
  • Automatic checkpointing, logging, resume from last checkpoint, early stopping

3. Model Implementation

All PyTorch modules, each with a clear forward/backward contract:

  • SupEncoder, DeepEncoder, CCEncoder (ConvNeXt-Tiny, 1-channel)
  • Phase4ProjectionHead (768 -> 512 -> 256)
  • Phase5TabularEncoder (clinical + biomarker vector -> 256)
  • Phase6CrossLayerAttention (3 tokens + identity embeddings -> 256 pooled)
  • Phase6LayerAwareOCTAEncoder (composes encoders + attention)
  • Phase7MaskEncoder (5-channel mask input -> 128-d)
  • Phase7CrossAttentionFusion (OCTA + tabular + mask -> 512-d)
  • Phase9PrototypeClassifier (512 -> 256 -> 128 -> N_classes + prototype logits)
  • Phase5ClinicalMaskTokenImputer (learned missing value tokens)
  • Phase8Generators (classical pseudo-mask generation, U-Net++ student)

4. Validation Pipeline

  • 5-fold cross-validation loop with patient-grouped outer/inner splits
  • Metric accumulation: macro-AUC, macro-F1, balanced accuracy, per-class metrics
  • 95% stratified patient-cluster bootstrap confidence intervals (1000 replicates)
  • Post-hoc temperature scaling on inner validation fold
  • MC-Dropout uncertainty: 30 forward passes, entropy, epistemic variance

5. Explainability Module

  • Grad-CAM per layer (Sup, Deep, CC) with predicted-class and true-label targets
  • Attention rollout: per-sample layer-dominance scores from cross-layer attention
  • SHAP or occlusion-based tabular attribution for clinical features
  • Integrated analysis: Grad-CAM vs. attention consistency summaries
  • Overlay heatmaps saved as PNGs + raw .npy arrays for later analysis

6. Experiment Manager (Lightweight)

  • Notebook-contained configuration dictionaries/cells
  • All hyperparameters logged to CSV/JSON at each phase
  • Reproducibility config with fixed random seeds (Python, NumPy, PyTorch, CUDA)
  • Deterministic PyTorch operations where available

Paper Components

The manuscript itself is not generated by the pipeline, but the following materials are produced to support writing:

  • Results tables: All metrics (primary + secondary) for the full model, all ablations, and all baselines, with bootstrap CIs, in CSV format ready to copy into LaTeX tables.
  • Figures: Confusion matrices, ROC curves, reliability diagrams, Grad-CAM overlays, attention dominance bar charts, SHAP beeswarm plots, missingness heatmap, biomarker boxplots, montage sheets -- all saved as publication-quality PNG/SVG.
  • Methodology section (draft): The Phase 18 contributions document provides draft paragraphs for the Methods and Discussion sections, written in academic style.
Publication tip: Start writing the Methods section early, while the implementation is fresh. The Phase docs (1-19) each contain the content you need for the corresponding subsection of your paper. Phase 18 (Contributions) maps directly to the Contributions paragraph in your Introduction and the Discussion of novelty.

Reproducibility Checklist

Before submission, verify:

  • All random seeds are fixed and logged (Python, NumPy, PyTorch, CUDA).
  • Cross-validation fold assignments are serialized and can be reloaded exactly.
  • Every hyperparameter used in every experiment is recorded in a CSV/JSON log.
  • The main.ipynb notebook is self-contained (no external .py files).
  • All output artifacts (CSVs, PNGs, JSONs, .pt checkpoints) have clear directory structure under outputs/.
  • Phase 14 metrics can be recomputed from Phase 12 outer-fold predictions alone (no need to retrain).
  • Ablation and baseline experiments are documented with their exact configuration deltas from the full model.
  • Hardware and software environment (GPU, CUDA version, PyTorch version, package versions) are recorded.

Glossary for Beginners

Artifact
Any file produced by the pipeline: a CSV table, a PNG image, a JSON config, a PyTorch checkpoint (.pt), a text log. These are the raw outputs of each phase.
Checkpoint
A saved snapshot of a model's weights (and optionally optimizer state, epoch number, etc.) that allows training to be paused and resumed, or the model to be loaded for inference later.
Reproducibility
The ability for an independent researcher to obtain the same results by following the same methods. In practice, this means providing all code, data paths, hyperparameters, random seeds, and environment specifications.
Bootstrap Confidence Interval
A method for estimating the uncertainty of a metric by repeatedly resampling the data (with replacement) and recomputing the metric each time. The 2.5th and 97.5th percentiles of the bootstrap distribution form the 95% CI.
