Fusion means the model learns from retinal images and clinical context together.

In this project, fusion does not mean dumping a spreadsheet next to pixels and hoping for the best. The plan is to first learn a compact representation from each retinal layer, learn a compact representation from the clinical variables, then merge those learned features into one shared pathway that produces class probabilities, uncertainty, and explanations.

What goes in

Three OCTA layers per eye, plus age, sex, BMI, smoking, hypertension, diabetes, dyslipidemia, and related clinical fields.

Where fusion happens

After the image and clinical branches each create learned feature vectors. This is feature-level fusion, also called intermediate fusion.

Why it exists

The same retinal pattern can mean different things depending on patient context. Clinical data helps the model interpret what the image pattern likely means.

Short answer

The planned model has two streams. One stream reads the three OCTA images. The other reads the clinical variables. Those streams meet in a fusion block, then continue as one prediction pathway.

Overview

What “fusion” means in plain language

Imagine a clinician looking at a scan and a chart at the same time. The scan shows what the eye looks like. The chart shows who the patient is and what risk factors they carry. Fusion is the model version of that joint reasoning.

In machine learning terms, each input type is first encoded into a numerical summary called a feature vector. Those vectors are then combined into a shared representation that is easier for the classifier to use than raw inputs.

This project is aiming for intermediate multimodal fusion. Raw pixels stay in the image branch. Raw tabular values stay in the clinical branch. Fusion happens after modality-specific encoders have transformed them into 256-dimensional learned representations.

The model is trying to classify disease cohorts from retinal OCTA. The hypothesis is that disease signal is spread across retinal depth and patient context, not confined to one image or one spreadsheet field.

Image only

Useful when the scan alone contains enough signal. Weak when the same visual pattern appears in different clinical settings.

Clinical only

Useful for broad risk estimation. Weak when retinal microvascular texture carries signal that the chart does not capture.

Fused model

Lets the model ask, “given this retinal pattern and this patient context, which disease cohort is most plausible?”

One sentence summary

Fusion is the step where the model turns “what the eye looks like” and “what we know about the patient” into one joint decision.

Step 1, inputs

The model sees three OCTA views of the same eye: superficial vascular complex, deep vascular complex, and choriocapillaris. It also sees clinical variables from the spreadsheet.

These are different kinds of evidence. Images show structure. Clinical variables show patient context.

The dataset object planned in phase 2 returns each layer separately, plus clinical_features and clinical_mask. That design choice already prepares the repo for fusion later.

Step 2, image branch

Each retinal layer gets its own encoder. That matters because SVC, DVC, and CC do not look the same biologically or visually.

After each encoder, the model has one 256-dimensional token per layer. A cross-layer attention block then learns how the layers relate to each other.

This is already one kind of fusion: image-with-image fusion across retinal depth. The plan calls this F_octa, the layer-aware OCTA representation.

Step 3, clinical branch

The clinical branch converts tabular variables into one compact clinical representation.

Missing values are not filled with a simple average. The model gets a signal that the value is missing.

The plan uses learnable missing-value mask tokens. That lets the model treat “not measured” as information instead of noise.

Step 4, fusion

This is the key point. The retinal representation and the clinical representation are merged into one shared state before classification.

The plan uses cross-attention, where the OCTA representation is conditioned by the clinical representation. In plain language, the model asks how the retinal evidence should be read for this specific patient context.

Because the plan compresses clinical data into a single 256-d token, this cross-attention behaves more like learned conditioning than full token-by-token attention. Conceptually it is still modality-aware fusion.

Step 5, output

The fused representation goes to the classifier head. That head predicts disease cohort probabilities.

The planned output also includes uncertainty, calibration, and explainability. So the model should say not only what it predicts, but also how certain it is and what evidence drove the prediction.

Fusion choices

Why this project uses intermediate fusion

There are three common ways to combine modalities. This project intentionally picks the middle option.

“Intermediate” means separate learning first, joint reasoning second. For this dataset, that is usually the safest compromise between flexibility, interpretability, and data efficiency.
Fusion style What it does Why it is weaker or stronger here
Early fusion Combine raw inputs very early. Hard here because pixels and clinical values are very different data types.
Late fusion Let each modality predict on its own, then average or vote. Easy to build, but weaker at learning nuanced interactions between modalities.
Intermediate fusion Learn modality-specific features first, then combine the learned features. This is the planned choice. It lets the image branch stay image-aware and the clinical branch stay clinically meaningful before they meet.
Practical effect The clinical branch does not need to learn edge detectors. The image branch does not need to interpret missing spreadsheet fields directly. Each branch solves the part of the problem it is best suited for, then the fusion block models cross-modal dependence.
Deep dive

Architecture overview, from basic to advanced

The same architecture can be understood at several levels. Start with the short explanations. Open the advanced notes when you want the design logic.

The project goal is not only prediction accuracy. It also aims for uncertainty, calibration, and explainability that clinicians can inspect.

1. Separate encoders for the three retinal layers

Each OCTA layer shows a different depth of retinal vasculature. So each gets its own image encoder.

The planned backbone is ConvNeXt-Tiny, pretrained on ImageNet, then adapted with OCTA SimCLR. After pooling, each branch produces a 256-d token.

Why not share one encoder across all layers?

The plan argues that the layers reflect different physiological compartments and different textures. Shared weights may force one filter bank to fit all three patterns, which can wash out layer-specific signal.

2. Cross-layer attention before multimodal fusion

Before mixing in the clinical table, the model first asks how the three retinal layers relate to each other.

Three layer tokens enter a self-attention block with learnable layer-identity embeddings. The output is pooled into one OCTA summary vector, F_octa.

Why layer-identity embeddings matter

Without them, self-attention sees three anonymous tokens. The model would know there are three things, but not reliably know which one is SVC, DVC, or CC.

3. Clinical encoder with explicit missingness handling

The model does not want a blank cell to quietly become a fake average patient.

Continuous values are z-scored, binary values are encoded, ordinal values like smoking can be embedded, and missingness is carried forward via masks or learned tokens.

Why that matters clinically

Missing data is often not random. A test that was not measured can itself carry information about workflow, patient profile, or chart completeness.

4. Multimodal fusion block

This is where image evidence and clinical evidence become one shared judgment.

The plan uses a cross-attention-style block, then concatenates fused image context with the clinical vector and projects the result into a shared 512-d state, H.

Why use attention instead of simple concatenation?

Concatenation always gives each modality the same structural role. Attention lets the model condition image interpretation on patient context more selectively.

5. Classification, prototypes, and uncertainty

The final layer predicts the disease cohort. The plan also wants the model to know when it is unsure.

The classifier head is paired with prototype-augmented logits, MC dropout, and temperature scaling so probabilities are more useful clinically.

Why uncertainty matters

Clinical models should not behave like overconfident black boxes. High-uncertainty cases can be flagged for manual review instead of being treated as equally reliable.

6. Explainability after fusion

The plan wants to show what parts of the retinal images mattered, and what clinical variables mattered.

Grad-CAM is planned for each image branch, attention analysis for layer dominance, and SHAP for clinical feature attribution.

Why this matters for trust

If the model says “this class is likely because the deep layer pattern mattered and diabetes strongly shifted interpretation,” that is more reviewable than a raw probability alone.

Advanced nuance

There are really two fusion moments in the planned architecture. First, fusion across retinal layers creates F_octa. Second, fusion across modalities combines F_octa with the clinical representation. The project therefore uses both intra-image fusion and multimodal fusion.

Why this architecture exists

What we are trying to create, and why this design is sensible

The architecture follows the biology, the data format, and the practical limits of a medium-sized medical dataset.

Put simply: keep the parts separate when they are fundamentally different, then merge them when the model has something meaningful to merge.

Reason 1, retinal depth matters

SVC, DVC, and CC capture different vascular compartments. Disease may leave a stronger signal in one layer than another, or in the relationship between layers.

Reason 2, context changes interpretation

A retinal pattern in a diabetic patient may mean something different from a visually similar pattern in a patient without diabetes or hypertension.

Reason 3, missingness must be handled honestly

Clinical tables are rarely perfect. The architecture is designed so absent values do not silently become misleading pseudo-measurements.

Reason 4, the dataset is not huge

The plan avoids giant end-to-end multimodal transformers from raw inputs. It uses compact feature spaces and staged training to reduce overfitting risk.

Glossary

Core terms, translated

If you know only basic AI/ML, these are the terms that matter most on this page.

The most important mental model: an encoder turns raw data into a compact summary that later parts of the model can reason over more easily.
Feature
A learned numerical pattern the model thinks is useful, such as vessel density texture or a compact summary of age and risk factors.
Encoder
The part of the model that reads an input and converts it into a feature vector.
Token
A vector treated as one item in attention. In this project, each retinal layer becomes one token after projection.
Attention
A mechanism that learns which other tokens matter when interpreting one token.
Cross-attention
Attention where one modality is conditioned by another modality. Here, retinal evidence is conditioned by clinical context.
Fusion
The step where separate sources of evidence are merged into one shared representation used for prediction.
Representation
The internal numerical summary the model carries forward after encoding.
Calibration
How well predicted probabilities match real-world correctness. A calibrated 80% confidence score should be right about 80% of the time.
Final takeaway

The architecture story in one paragraph

We are building a multimodal OCTA classifier that first learns each retinal layer on its own, then learns how the layers relate, then uses clinical context to refine that retinal interpretation, and finally produces a disease prediction with uncertainty and explainability. The key idea is that retinal images and clinical variables should stay separate long enough to be understood well, then join before the final decision so the model can reason with both at once.

Repo truth today: the groundwork is implemented, especially the data pipeline and OCTA pretraining. The actual fusion block is the next major architecture step.
