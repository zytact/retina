Measuring What Matters

A medical AI model is only as good as the evidence that it works. This phase builds a rigorous evaluation framework: how we measure performance, how we know those measurements are reliable, and how we compare models fairly. The goal is to produce publication-quality numbers that clinicians and reviewers can trust.

Why Evaluation Matters

A model that looks impressive on one metric may fail badly on another. Rigorous evaluation is what separates a real contribution from a lucky run.

Imagine you build a model that predicts whether a patient has coronary artery disease from their retinal OCTA scan. You run it on your test set and get 92% accuracy. Sounds great, right? But what if 90% of your patients are healthy controls? A model that predicts "healthy" for everyone gets 90% accuracy without learning anything. Accuracy alone tells you very little when classes are imbalanced.

This is why medical AI evaluation needs multiple metrics, each answering a different question. The Retina CAD project classifies patients into four cohorts with varying prevalence:

The smallest cohort (MRCC) may be 5-10x smaller than the largest (RETINORM). Any single-number metric will be dominated by the majority class. That is why we need a suite of metrics.

Phase 14 addresses four fundamental questions about the model's performance:

How well does the model discriminate between classes?

Answered by ROC-AUC, PR-AUC, and per-class sensitivity/specificity. These metrics measure whether the model can separate diseased from healthy patients, and one disease from another.

How reliable are the predicted probabilities?

Answered by calibration metrics (ECE, Brier score) and reliability diagrams. A model can have great discrimination (it ranks patients correctly) but poor calibration (it says "80% confident" when it is right only 60% of the time).

How uncertain is each prediction?

Answered by Monte Carlo Dropout uncertainty analysis. Some predictions are inherently uncertain (e.g., ambiguous scans, patients with atypical features). Identifying these allows the system to flag them for expert review rather than making a potentially wrong decision.

Is this model significantly better than an alternative?

Answered by statistical tests: McNemar's test for paired classification comparison and DeLong's test for AUC comparison. A 0.01 improvement in AUC might be real or it might be noise. Statistical tests tell us which.

This framework follows best practices from the biomedical ML literature and aligns with reporting standards for medical AI studies (e.g., TRIPOD-AI, STARD-AI). Every metric reported in future phases (ablation studies in Phase 15, baseline comparisons in Phase 16) uses this same evaluation pipeline, ensuring consistency across all results.

Bootstrap Confidence Intervals

Every single-number metric is incomplete. Confidence intervals tell you how much to trust it.

What is bootstrapping?

When you compute a metric (say, accuracy) on your test set, you get one number: 87.3%. But if you had a slightly different test set (different patients, different eyes), that number would change. How much? The bootstrap answers this question by simulating "what if we had a slightly different test set" over and over.

Here is how it works in practice:

  1. Start with your test set of N predictions (one per patient or per eye).
  2. Sample with replacement N items from your test set to create a "bootstrap sample." Because you sample with replacement, some patients appear multiple times and some are left out.
  3. Compute your metric (AUC, F1, etc.) on this bootstrap sample.
  4. Repeat 1,000 (or more) times.
  5. Take the 2.5th and 97.5th percentiles of the 1,000 computed values. This range is the 95% confidence interval.
Patient-level bootstrap, not eye-level. Because the dataset contains bilateral eyes (OD and OS) from the same patient, a simple eye-level bootstrap would treat both eyes as independent — violating the assumption that bootstrap samples are independent. The evaluation pipeline uses a patient-clustered bootstrap: it resamples patients (not eyes), then includes both eyes from sampled patients. This preserves within-patient correlation and produces valid confidence intervals.

Why this matters:

  • Without CIs, you cannot tell if an apparent improvement (e.g., AUC 0.87 vs. 0.85) is meaningful or just noise.
  • Bootstrapping makes no assumptions about the underlying distribution (unlike parametric methods that assume normality).
  • The CI width tells you the precision of your estimate. A wide CI means you need more data to be confident in the result.
  • Stratified bootstrap (resampling within each cohort) ensures minority classes are not accidentally omitted from some bootstrap samples.
How to read a bootstrap CI
Macro-F1 = 0.74
95% CI [0.67, 0.80]

This means: if we repeated the experiment 100 times with different patient samples, the Macro-F1 would fall between 0.67 and 0.80 in 95 of those experiments. The interval width (0.13) tells us there is moderate uncertainty. A width of 0.03 would mean high precision.

In the evaluation pipeline, every metric reported in the final results tables is accompanied by 95% bootstrap confidence intervals with 1,000 replicates. This applies to primary metrics, secondary metrics, calibration metrics, and per-class metrics alike.

Eye-Level vs. Patient-Level Metrics

Should we evaluate per eye or per patient? The answer depends on the question we are asking.

The RASTA dataset contains both eyes (OD and OS) for many patients. This creates a choice: do we report metrics at the eye level (each eye is a sample) or at the patient level (average both eyes into one prediction per patient)?

Eye-level metrics

Treat each eye as an independent sample. Both OD and OS appear as separate rows in the evaluation. This gives a larger N and higher statistical power. It answers: "How well does the model classify a single OCTA scan?"

Patient-level metrics

Average the softmax probabilities from OD and OS (when both are available) into a single patient-level prediction. This answers: "How well does the model classify a patient?" which is the clinically relevant question — clinicians treat patients, not individual eyes.

Both are reported in the evaluation bundle:

  • Eye-level metrics provide the primary analysis, matching how other OCTA studies typically report results. They use all available data and provide the most statistical power for detecting differences between models.
  • Patient-level metrics serve as the publication-facing and clinically oriented analysis. Averaging both eyes is a natural ensembling strategy: if the model agrees on both eyes, confidence increases; if the eyes disagree, the average reflects uncertainty.

Important: The bootstrap confidence intervals for patient-level metrics use patient-level resampling (resample patients, then average both eyes within each resampled patient). This preserves the within-patient correlation structure and produces valid intervals.

For bilateral eyes, patient-level predictions are formed by averaging OD and OS softmax probabilities before computing the final class prediction. For patients with only one eye available, that eye's prediction is used directly as the patient-level prediction.

Primary Metrics

Three metrics form the primary evaluation battery. Together, they tell us whether the model is genuinely discriminative, balanced across classes, and clinically useful.

Macro-averaged ROC-AUC

The ROC (Receiver Operating Characteristic) curve plots the true positive rate against the false positive rate as the classification threshold varies. The AUC (Area Under the Curve) summarizes the entire curve into a single number: the probability that a randomly chosen positive sample will be ranked higher than a randomly chosen negative sample.

Plain English interpretation: If you show the model one random patient from cohort A and one random patient from every other cohort combined, the AUC tells you how often the model gives a higher score to the correct cohort. An AUC of 0.5 means random guessing. An AUC of 1.0 means perfect ranking. For medical tests, AUC > 0.8 is generally considered good, and AUC > 0.9 is excellent.

Because we have four cohorts, we compute AUC in a one-vs-rest manner: for each cohort, treat it as the positive class and all other cohorts as the negative class. This gives four per-class AUCs. The macro-averaged ROC-AUC is the unweighted average of these four values.

Macro-averaged ROC-AUC

AUCmacro = (1/4) · ∑k=14 AUCclass k

Each per-class AUC is computed one-vs-rest. Macro-averaging gives equal weight to each class regardless of its prevalence. This is the standard for multi-class medical imaging studies and is robust to class imbalance.

Macro-F1 and Balanced Accuracy

While ROC-AUC measures ranking quality, Macro-F1 and Balanced Accuracy measure classification quality at a specific decision threshold (usually 0.5 after temperature scaling).

Macro-F1

The F1 score is the harmonic mean of precision and recall. For multi-class problems, we compute F1 per class and then average them (macro-averaging).

F1 score per class

Precisionk = TPk / (TPk + FPk)
Recallk = TPk / (TPk + FNk)
F1k = 2 · (Precisionk · Recallk) / (Precisionk + Recallk)

Macro-F1 = (1/4) · ∑k=14 F1k

Macro-F1 penalizes the model equally for poor performance on any class, regardless of size. A model that ignores MRCC entirely will have a low Macro-F1 even if overall accuracy is high.

Why Macro-F1 instead of Weighted F1?

Weighted F1 averages per-class F1 scores weighted by class size. This gives the largest class (RETINORM) the most influence. But we care most about whether the model works for the rare but clinically critical classes (MRCC). Macro-F1 gives MRCC the same weight as RETINORM, which is a harder test and more clinically honest.

Why not just accuracy?

Accuracy = (TP + TN) / Total. With class imbalance, accuracy is misleading. If RETINORM has 200 samples and MRCC has 20, a model that always predicts RETINORM achieves 200/220 = 91% accuracy while being clinically useless. Macro-F1 and balanced accuracy avoid this trap.

Balanced Accuracy

Balanced accuracy is the average of recall (sensitivity) across all classes. Unlike standard accuracy, it is immune to class imbalance:

Balanced Accuracy

Balanced Accuracy = (1/4) · ∑k=14 (TPk / (TPk + FNk))

This is simply the average of per-class recall. Random guessing gives 0.25 for 4 classes. A perfect model gives 1.0.

Per-Class Sensitivity and Specificity

For clinical applications, the overall macro-averaged metrics are not enough. We need to know: for each disease cohort, what fraction of patients with that condition does the model detect (sensitivity), and what fraction of patients without that condition does it correctly identify as not having it (specificity)?

Sensitivity (Recall / True Positive Rate)

Of all patients who truly have cohort X, how many did the model correctly identify? High sensitivity means fewer missed cases. For a screening tool, high sensitivity is critical: missing a case of CAD (MRCC) could have serious consequences.

Specificity (True Negative Rate)

Of all patients who do not have cohort X, how many did the model correctly identify as negative? High specificity means fewer false alarms. For a screening tool, low specificity leads to unnecessary follow-up tests and patient anxiety.

The sensitivity-specificity tradeoff. Increasing sensitivity (detecting more true cases) almost always decreases specificity (more false alarms). The optimal balance depends on the clinical use case. For screening, bias toward sensitivity. For confirmatory diagnosis, bias toward specificity. The ROC curve shows this tradeoff at every possible threshold.

In the final results table, per-class sensitivity and specificity are reported at the default decision threshold (after temperature scaling), with the option to adjust thresholds per class to optimize a clinically defined operating point (e.g., ensure ≥ 95% sensitivity for MRCC at the cost of reduced specificity).

Secondary Metrics

Secondary metrics provide additional perspective and diagnostic detail beyond the primary battery.

Confusion Matrix

The confusion matrix is the most direct way to see which classes the model confuses. It is a 4x4 grid where rows are true classes and columns are predicted classes. The diagonal shows correct predictions; off-diagonal cells show errors.

Hypothetical confusion matrix
Pred RETPred ORNPred FAMPred MRCC
True RET84673
True ORN862115
True FAM910587
True MRCC43530
Diagonal = correct. Higher is better. This matrix shows MRCC is most often confused with FAMILIPO, which makes clinical sense.

What to look for in a confusion matrix:

  • Which pairs are most confused? If the model often predicts ORNET when the truth is FAMILIPO, there may be shared vascular features that make them hard to distinguish.
  • Is one class absorbing most errors? If the largest class (RETINORM) gets many false positives, the model may be biased toward predicting the majority class.
  • Normalized version: The pipeline also produces a row-normalized confusion matrix (each row sums to 1.0), showing the proportion of each true class that falls into each predicted class. This is better for comparing across classes of different sizes.

PR-AUC (Precision-Recall AUC)

While ROC-AUC is the standard for balanced datasets, Precision-Recall AUC (PR-AUC) is more informative when classes are imbalanced — which is exactly the case in the RASTA dataset. PR-AUC focuses on the positive class (the minority) rather than the negative class (the majority).

ROC-AUC vs. PR-AUC: ROC-AUC plots TPR vs. FPR. Because FPR uses the large negative class in its denominator, it can make a model look better than it is when negatives vastly outnumber positives. PR-AUC plots precision vs. recall, both of which focus on the positive (minority) class. For imbalanced medical datasets, PR-AUC is often the more honest metric.

Macro-averaged PR-AUC is computed the same way as macro-averaged ROC-AUC: one-vs-rest per class, then averaged. The baseline for PR-AUC is the prevalence of the positive class (e.g., if MRCC is 15% of the data, a random classifier gets PR-AUC = 0.15).

Top-2 Accuracy

For a 4-class problem, top-2 accuracy measures whether the correct class is in the model's top 2 predictions. This is a more forgiving metric that is useful when some classes are genuinely hard to distinguish (e.g., FAMILIPO vs. MRCC both involve cardiovascular risk). If the model consistently puts the correct class in its top 2, it means the ranking is sensible even if the top-1 prediction is wrong.

Top-2 accuracy is particularly relevant for clinical triage: if the model lists "MRCC or FAMILIPO" as the top 2, a clinician knows to investigate cardiovascular risk even if the exact subcategory is uncertain.

Calibration — How Much Should You Trust That 80%?

A model that says "80% confident" should be right 80% of the time. Calibration metrics tell us whether the probabilities are trustworthy.

Imagine two models that both achieve 85% accuracy on the same test set. Model A says "I am 95% confident" when it is right and "I am 60% confident" when it is wrong. Model B says "I am 85% confident" for every prediction. Model B is perfectly calibrated (on average), even though it provides less informative individual probabilities. Model A is miscalibrated: it is overconfident when it is right and underconfident when it is wrong.

For medical decision-making, calibration matters because:

  • Clinical workflow integration — If the model says a patient has a 90% probability of MRCC, a clinician may decide to prioritize that patient for cardiac workup. If this "90%" is actually 60% in reality, patients are being mismanaged.
  • Screening thresholds — Many screening programs use probability thresholds (e.g., refer if CAD risk > 20%). Miscalibrated probabilities lead to incorrect referral decisions.
  • Regulatory acceptance — Calibration is part of the "analytical validity" required by regulators. A model with good AUC but poor calibration may not pass regulatory review for clinical use.

Expected Calibration Error (ECE)

ECE is the most common metric for evaluating calibration. It works by:

  1. Binning predictions by confidence. For example, create 10 bins: [0-10%], [10-20%], ..., [90-100%].
  2. For each bin, compute the average predicted probability and the fraction of positives (accuracy).
  3. ECE is the weighted average of |accuracy - confidence| across bins.
Expected Calibration Error (10 bins)

ECE = ∑b=110 (nb / N) · |acc(b) - conf(b)|

Where nb is the number of predictions in bin b, N is the total number of predictions, acc(b) is the fraction of correct predictions in bin b, and conf(b) is the average predicted confidence in bin b. Lower ECE is better. ECE = 0 means perfect calibration.
Reading ECE values
ECEInterpretation
< 0.02Excellent calibration
0.02 – 0.05Good calibration
0.05 – 0.10Moderate; may need recalibration
> 0.10Poor; probabilities unreliable

Reliability diagram: The evaluation pipeline also generates a reliability diagram — a plot with predicted confidence on the x-axis and actual accuracy on the y-axis. Perfect calibration is the diagonal line y = x. Points above the diagonal mean the model is underconfident; points below mean it is overconfident.

The diagram is generated per class, so you can see if calibration varies across cohorts.

Brier Score

The Brier score is the mean squared error between predicted probabilities and one-hot encoded true labels. It captures both discrimination and calibration in a single number.

Multi-class Brier Score

Brier = (1/N) · ∑i=1N ∑k=14 (pik - yik)2

Where pik is the predicted probability for sample i and class k, and yik is 1 if the true label is k, 0 otherwise. Lower is better. The Brier score ranges from 0 (perfect) to 2 (worst for 4 classes).

The Brier score can be decomposed into three components: uncertainty (aleatory), reliability (calibration), and resolution (discrimination). This decomposition helps diagnose why the Brier score is high: is it because the model is poorly calibrated, or because the classes are inherently overlapping (high uncertainty)?

Temperature scaling. Before computing calibration metrics, the raw model logits are passed through a learned temperature parameter T > 0. Softmax(logits / T) produces calibrated probabilities. The temperature is learned on the inner validation split by minimizing negative log-likelihood, then applied to the held-out test predictions. This is post-hoc calibration — it does not change the model's ranking ability (AUC is unaffected), only the confidence values.

Uncertainty Evaluation — Knowing When to Say "I Don't Know"

A good medical AI knows its limits. Uncertainty quantification identifies predictions that should not be trusted.

The Retina CAD model uses Monte Carlo Dropout (MC-Dropout) at inference time. During training, dropout layers randomly zero out neurons. At inference, normally dropout is turned off. But with MC-Dropout, we leave dropout on and run 30 forward passes. Each pass produces slightly different predictions. The variability across these 30 passes measures the model's epistemic uncertainty — uncertainty due to limited training data or ambiguous inputs.

Input image → Run 1 (dropout on) → Prediction 1 ⋮ Run 30 (dropout on) → Mean & Variance

From the 30 forward passes, we compute:

  • Mean prediction: The average softmax probability across all 30 passes. This is the final prediction.
  • Epistemic variance: The variance of the probabilities across the 30 passes, per class. High variance means the model is uncertain about that class's probability.
  • Predictive entropy: A single number summarizing overall uncertainty: H = -∑ pk log(pk). Higher entropy = more uncertain.

Entropy vs. Accuracy Analysis

The key validation question is: Are uncertain predictions also less accurate? If the model's uncertainty estimates are meaningful, then predictions with high entropy should have lower accuracy. The evaluation pipeline tests this by:

  1. Sorting all predictions by entropy (low to high).
  2. Dividing into quartiles (0-25%, 25-50%, 50-75%, 75-100% entropy).
  3. Computing accuracy within each quartile.
Expected pattern for a well-calibrated uncertainty model:
Entropy quartileAccuracyInterpretation
Lowest (most confident)~95%High-confidence predictions are very reliable
Second~85%Still reliable
Third~70%Moderate uncertainty
Highest (most uncertain)~40%Should be flagged for manual review

Coverage Analysis

Coverage analysis asks: "What percentage of correct predictions fall in the top confidence bin?" If the top 30% most confident predictions contain 90% of all correct predictions, the model's confidence ranking is informative. This is a practical metric for clinical deployment: you could set a confidence threshold, automatically accept predictions above it, and refer the rest for manual review.

Clinical workflow implication: With a coverage of 80% at 95% accuracy, you could automate 80% of cases with high confidence and have a clinician review the remaining 20%. This reduces workload while maintaining safety.

Statistical Comparison — Is Model A Really Better Than Model B?

When two models differ by 0.01 in AUC, is that a real improvement or just noise? Statistical tests provide the answer.

Phase 14 provides two statistical tests for comparing models. These are essential for ablation studies (Phase 15) and baseline comparisons (Phase 16), where we need to know whether adding or removing a component makes a statistically significant difference.

McNemar's Test

McNemar's test compares two models on a paired classification task. It answers: "Do the two models make different errors?" It uses only the samples where the models disagree.

McNemar's Test

Contingency table:

Model B correctModel B wrong
Model A correctn00n01
Model A wrongn10n11
χ2 = (|n01 - n10| - 1)2 / (n01 + n10)

The null hypothesis is that the two models have equal error rates. If p < 0.05 (after Bonferroni correction for multiple comparisons), we reject the null and conclude one model makes significantly fewer errors. McNemar is a paired test, meaning the same test samples are used for both models.

Interpreting McNemar results:

  • If n01 > n10, Model A makes more errors on samples that Model B gets right → Model B is better.
  • If the p-value is small (< 0.05), the difference is statistically significant.
  • McNemar only tells you if the error rates are different, not which model is better on specific classes. For class-level analysis, use per-class metrics.
When to use McNemar

McNemar is the default test for comparing two classification models on the same test set. It is preferred over a simple accuracy comparison because it accounts for the paired nature of the data (same samples, two models). It is also distribution-free (no assumptions about normality).

DeLong Test for AUC Comparison

When comparing AUC values between two models, the DeLong test (DeLong et al., 1988) is the standard method. It tests whether the difference between two correlated ROC-AUC values is statistically significant.

The DeLong test uses the theory of U-statistics to estimate the covariance matrix of the two AUC estimates, accounting for the fact that the same test samples are used for both models (paired/matched data). It produces a z-statistic and p-value for the null hypothesis that the two AUCs are equal.

Multiple Comparison Correction

When running multiple statistical tests (e.g., comparing the full model against 10 baselines and 15 ablations), the chance of at least one false positive increases dramatically. The pipeline applies Bonferroni correction: the significance threshold α = 0.05 is divided by the number of tests. If we run 25 comparisons, the corrected threshold is 0.05 / 25 = 0.002. Only p-values below this threshold are considered significant.

Effect Size (Cohen's h)

Statistical significance is not the same as practical significance. A tiny effect can be statistically significant with enough data. Cohen's h measures the effect size for comparing two proportions or AUC values:

Cohen's h for AUC comparison

h = 2 · arcsin(√AUC1) - 2 · arcsin(√AUC2)

Convention: h ≥ 0.2 = small effect, h ≥ 0.5 = medium effect, h ≥ 0.8 = large effect. Effect size tells you whether a statistically significant difference is also practically meaningful.

The evaluation pipeline reports both p-values and effect sizes for all pairwise model comparisons, ensuring that conclusions are based on both statistical and practical significance.

How to Read the Results

A practical guide to interpreting the evaluation output and distinguishing meaningful findings from noise.

The results bundle

When the evaluation pipeline runs, it produces a structured bundle under outputs/phase14_evaluation/:

  • fold_<k>_eye_metrics.csv — Per-fold eye-level metrics with bootstrap CIs.
  • fold_<k>_patient_metrics.csv — Per-fold patient-level metrics with bootstrap CIs.
  • fold_<k>_calibration.csv — ECE, Brier score, reliability bins.
  • fold_<k>_confusion_matrix.csv — Raw and normalized confusion matrices.
  • fold_<k>_mc_dropout.csv — MC-Dropout uncertainty predictions with entropy and per-class variance.
  • cross_validation_summary.csv — Mean ± std across 5 folds for all primary and secondary metrics.

How to interpret the numbers

Macro-AUC is 0.85. Is that good?

For a 4-class medical image classification task with moderate class imbalance and a medium-sized dataset (~476 eyes), 0.85 is competitive. It means the model ranks about 85% of positive samples higher than negative samples, averaged across all four one-vs-rest comparisons. Compare this to the baselines (Phase 16) to see if the multimodal approach adds value over simpler models.

Macro-F1 is 0.68 but weighted F1 is 0.78. Why the gap?

This gap means the model performs better on larger classes (RETINORM) than on smaller ones (MRCC). Macro-F1 penalizes poor MRCC performance equally, while weighted F1 lets RETINORM's size mask it. The gap itself is diagnostic: if it is large, minority-class improvement should be a priority.

ECE is 0.08 after temperature scaling. What does this mean?

An ECE of 0.08 means the average gap between predicted confidence and actual accuracy is 8 percentage points. For example, predictions where the model says "70% confident" are actually correct about 62% of the time on average. This is moderate calibration — usable for ranking but not yet reliable enough for threshold-based clinical decisions without further calibration improvement.

MC-Dropout entropy is high for a specific patient. What should I do?

High entropy means the model is uncertain about this patient. In a clinical workflow, this patient would be flagged for expert review. Review the Grad-CAM heatmaps (Phase 13) for this patient to see what the model was looking at. The uncertainty signal is a feature, not a bug — it tells you when to trust the model and when to be cautious.

DeLong test says p = 0.04 but Bonferroni threshold is 0.002. Is it significant?

With Bonferroni correction for 25 comparisons, no, p = 0.04 is not significant after correction. The observed AUC difference is likely due to chance. This is why multiple comparison correction is essential: without it, about 1 in 20 tests would be falsely called significant just by chance.

Remember: No single metric tells the full story. ROC-AUC tells you about ranking. F1 tells you about classification at a specific threshold. ECE tells you about probability reliability. Entropy analysis tells you about per-sample uncertainty. Together, they provide a complete picture. Always look at the full evaluation bundle, not just one number.
